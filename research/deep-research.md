# Technical Project: Local RAG Demo

## 1. Идея проекта

Делаем простой локальный RAG-сервис для школьного хакатона.

Пользователь загружает несколько небольших документов и задает по ним вопросы в чате. Система находит подходящие фрагменты текста, добавляет их в контекст и формирует ответ локальной LLM-моделью.

Проект должен запускаться на обычном ноутбуке без GPU, поэтому архитектура и стек специально упрощены и жестко лимитированы по ресурсам.

## 2. Цель

Показать эволюцию проекта от идеи до рабочего прототипа и собрать минимальный локальный AI-сервис, который:

* работает без облака (полностью оффлайн);
* запускается одной-двумя командами через Docker Compose;
* использует Open WebUI как интерфейс пользователя;
* использует `llama.cpp` для локального инференса (чат и эмбеддинги);
* поддерживает простой RAG по небольшим документам.

## 3. Базовые предположения

* **Дата проектирования:** март 2026.
* **Целевая среда:** ноутбуки школьников без GPU (8-16 ГБ RAM).
* **Документы для demo:** небольшие `.txt`, `.md`, простые `.pdf`.
* **Нагрузка:** один пользователь (один запрос в моменте).
* **Приоритет:** надежность запуска, предсказуемое потребление памяти и простота, а не высокая скорость или сложные пайплайны.

## 4. Выбранный стек

* **Open WebUI** — веб-интерфейс, загрузка документов, чат, встроенный векторный поиск (ChromaDB).
* **llama.cpp server** (2 инстанса) — локальный OpenAI-compatible backend для chat и embeddings.
* **Docker Compose** — оркестрация контейнеров.
* **Bash/curl** — для предварительного скачивания весов моделей.

## 5. Выбранные модели и квантование

Для работы на слабых CPU критически важен выбор правильного уровня квантования. Мы используем формат GGUF с квантованием **Q4_K_M** — это лучший баланс между сохранением интеллекта модели и потреблением RAM.

### 5.1. Основная LLM (Chat)
**Модель:** `Qwen/Qwen3-1.7B-Instruct-GGUF` (файл: `*q4_k_m.gguf`)
* **Почему:** Актуальная компактная модель, отлично понимает русский язык, следует инструкциям. В формате Q4_K_M занимает около 1.2–1.5 ГБ оперативной памяти.

### 5.2. Embedding-модель (RAG)
**Модель:** `Qwen/Qwen3-Embedding-0.6B-GGUF` (файл: `*q4_k_m.gguf`)
* **Почему:** Легковесная мультиязычная модель для векторизации. На CPU работает достаточно быстро. Занимает менее 0.5 ГБ RAM.

## 6. Архитектура

**Логическая схема:**
```text
User 
 └──> Open WebUI (порт 3000)
       ├──> llama-chat (порт 8080)[Генерация ответов]
       └──> llama-embed (порт 8081)[Создание векторов для документов и запросов]
```

**Поток данных (RAG):**
1. Пользователь загружает PDF в Open WebUI.
2. Open WebUI парсит текст, бьет на чанки и отправляет их в `llama-embed`.
3. Полученные векторы сохраняются во встроенную ChromaDB (внутри контейнера Open WebUI).
4. Пользователь пишет вопрос в чат.
5. Вопрос векторизуется через `llama-embed`, система ищет релевантные чанки в БД.
6. Найденный текст и вопрос отправляются в `llama-chat` с RAG-промптом.
7. Модель генерирует ответ.

## 7. Требования к инфраструктуре и ресурсам (Критично для CPU-only)

Чтобы ноутбук не завис при генерации, необходимо жестко ограничить ресурсы в `docker-compose.yml`:

1. **Ограничение контекста (Context Size):** Для чат-модели передать флаг `-c 4096`. Этого хватит для RAG (пару чанков текста + вопрос), но не позволит модели съесть всю RAM.
2. **Ограничение потоков (Threads):** Передать флаг `--threads 4`. Это оставит свободные ядра CPU для работы самой ОС и базы данных Open WebUI.
3. **Отключение встроенных моделей Open WebUI:** В `.env` необходимо явно прописать использование OpenAI-совместимого эндпоинта для эмбеддингов, чтобы Open WebUI не пытался качать тяжелые sentence-transformers модели на Python.

## 8. Этапы реализации

### Этап 1. Скрипт подготовки (download_models.sh)
* Создание папок `./models` и `./data`.
* Скачивание двух `.gguf` файлов с HuggingFace (через `curl` или `wget`) в папку `./models`.
* *Это делается до запуска Docker, чтобы избежать таймаутов.*

### Этап 2. Инфраструктура (Docker Compose & .env)
* Написание `docker-compose.yml` с тремя сервисами: `open-webui`, `llama-chat`, `llama-embed`.
* Написание `.env` с правильными URL (например, `OPENAI_API_BASE_URL=http://llama-chat:8080/v1` и переменными для `RAG_EMBEDDING_ENGINE`).

### Этап 3. Документация и запуск
* Написание `README.md` с пошаговой инструкцией: как скачать модели, как запустить контейнеры, как настроить системный RAG-промпт в UI.

## 10. Критерии готовности MVP

MVP считается готовым, если:
1. Выполняется `bash download_models.sh` (модели скачиваются).
2. Выполняется `docker compose up -d` (все три контейнера поднимаются, ноутбук не "умирает" от нехватки памяти).
3. Open WebUI доступен на `localhost:3000`.
4. В логах Open WebUI нет попыток скачать `sentence-transformers` (эмбеддинги идут в `llama-embed`).
5. Загрузка текстового документа проходит успешно.
6. Вопрос по документу генерирует осмысленный ответ на русском языке на основе контекста.

## 11. Риски и их митигация

* **Риск:** Ноутбук зависает при генерации. 
  **Решение:** Жестко зафиксированы параметры `--threads 4` и `-c 4096`.
* **Риск:** Open WebUI не видит llama.cpp embeddings.
  **Решение:** Использовать стандартный OpenAI-совместимый эндпоинт llama.cpp и пробросить его через переменные `RAG_OPENAI_API_BASE_URL`.
* **Риск:** Маленькая модель (1.7B) отвечает на английском при RAG-запросах.
  **Решение:** Явное указание пользователю прописать русскоязычный системный RAG-промпт в настройках Open WebUI.

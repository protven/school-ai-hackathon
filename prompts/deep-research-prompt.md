# Deep Research Prompt

Нужно провести исследование и подготовить основу для технического проекта.

## Контекст

Есть идея учебного проекта для школьного хакатона.

Проект: простой локальный RAG-сервис, который запускается на обычном ноутбуке без GPU.

Пользователь загружает несколько небольших документов и задает по ним вопросы в чате. Система находит релевантные фрагменты текста и формирует ответ с учетом найденного контекста.

Проект должен быть максимально простым, воспроизводимым и понятным. Это не production-система, а учебный демонстрационный прототип.

## Цель исследования

Нужно подготовить практическую и технически обоснованную основу для следующего шага — написания технического проекта.

## Что нужно исследовать

1. Насколько реалистично собрать такой локальный RAG-сервис на обычном ноутбуке без GPU.
2. Какой минимальный стек лучше использовать для такого сценария.
3. Подходит ли связка Open WebUI + llama.cpp для простого локального RAG.
4. Какие компактные модели на текущий момент лучше подходят:
   - для локальной chat model;
   - для embedding model;
   - с учетом марта 2026;
   - с учетом CPU-only запуска;
   - с учетом русскоязычных и мультиязычных сценариев.
5. Какие ограничения и риски есть у такого решения.
6. Какие решения нужно сознательно не включать в MVP, чтобы не усложнять проект.
7. Какой минимальный состав сервисов нужен.
8. Какие форматы документов разумно поддержать в первой версии.
9. Какие реалистичные границы MVP для такого проекта.

## Ожидаемый результат

Нужен структурированный исследовательский документ, который можно использовать как основу для технического проекта.

В результате должны быть:

- краткое описание рекомендуемого подхода;
- рекомендуемый стек;
- рекомендуемые модели;
- аргументация выбора;
- список ограничений;
- список рисков;
- рекомендуемые границы MVP;
- вывод, подходит ли идея для реализации в учебном формате.

## Требования к ответу

- Не предлагать лишнюю инфраструктуру.
- Не усложнять архитектуру без необходимости.
- Исходить из CPU-only ноутбуков.
- Давать рекомендации не “вообще лучшие”, а лучшие в контексте этого проекта.
- Указывать компромиссы и ограничения.
- Писать технически, но понятно.

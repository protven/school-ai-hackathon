# Technical Project: Local RAG Demo

## 1. Идея проекта

Делаем простой локальный RAG-сервис для школьного хакатона.

Пользователь загружает несколько небольших документов и задает по ним вопросы в чате. Система находит подходящие фрагменты текста, добавляет их в контекст и формирует ответ локальной LLM-моделью.

Проект должен запускаться на обычном ноутбуке без GPU, поэтому архитектура и стек специально упрощены и жестко ограничены по ресурсам.

## 2. Цель

Показать эволюцию проекта от идеи до рабочего прототипа и собрать минимальный локальный AI-сервис, который:

- работает полностью локально, без обращения к облачным API;
- запускается одной-двумя командами через Docker Compose;
- использует Open WebUI как пользовательский интерфейс;
- использует `llama.cpp` для локального инференса;
- поддерживает простой векторный поиск (RAG) по небольшим документам.

## 3. Базовые предположения

- **Дата проектирования:** март 2026.
- **Целевая среда:** ноутбуки школьников без дискретных GPU, примерно 8–16 ГБ RAM.
- **Документы для demo:** небольшие `.txt`, `.md`, простые `.pdf`.
- **Нагрузка:** один пользователь, один запрос в моменте.
- **Приоритет:** надежность запуска, предсказуемое потребление памяти и простота, а не высокая скорость генерации или сложные data-пайплайны.

## 4. Выбранный стек

- **Open WebUI** — веб-интерфейс, парсинг документов, чат, встроенная векторная БД.
- **llama.cpp server** — легковесный OpenAI-compatible backend для chat и embeddings.
- **Docker Compose** — оркестрация контейнеров.
- **Bash / curl / wget** — скрипты для предварительного скачивания весов моделей.

## 5. Выбранные модели и квантование

Для слабых CPU критичен выбор разумного квантования. На первом этапе фиксируем формат GGUF с квантованием **Q4_K_M** как оптимальный компромисс между качеством модели и потреблением оперативной памяти.

### 5.1. Основная LLM (Chat)

**Модель:** `Qwen/Qwen3-1.7B-Instruct-GGUF`  
**Файл:** вариант с `q4_k_m`

Почему выбрана:
- компактная;
- хорошо работает в инструктивном диалоге;
- подходит для локального CPU-only demo;
- дает разумный баланс качества и требований к ресурсам.

### 5.2. Embedding-модель (RAG)

**Модель:** `Qwen/Qwen3-Embedding-0.6B-GGUF`  
**Файл:** вариант с `q4_k_m`

Почему выбрана:
- легкая embedding-модель;
- мультиязычная;
- подходит для локального поиска по документам;
- реалистична для обычных ноутбуков.

## 6. Архитектура

### 6.1. Логическая схема

```text
User
 └──> Open WebUI (порт 3000)
       ├──> llama-chat  (порт 8080)  — генерация ответов
       └──> llama-embed (порт 8081)  — векторизация документов и запросов
```

### 6.2. Поток работы

1. Пользователь открывает Open WebUI.
2. Загружает документ.
3. Open WebUI извлекает текст и разбивает его на чанки.
4. Чанки отправляются в `llama-embed` для получения embeddings.
5. Векторы сохраняются во внутреннем хранилище Open WebUI.
6. Пользователь задает вопрос.
7. Вопрос векторизуется через `llama-embed`.
8. Система ищет релевантные чанки.
9. Найденный контекст передается вместе с вопросом в `llama-chat`.
10. Модель формирует ответ.

## 7. Границы MVP

### 7.1. Что входит в MVP

В первую версию входят только следующие возможности:

- локальный запуск на ноутбуке без GPU;
- запуск через Docker Compose;
- Open WebUI как единый интерфейс;
- локальная chat model через `llama.cpp`;
- локальная embedding model через `llama.cpp`;
- загрузка небольших документов;
- простой сценарий RAG: вопрос по документу → поиск релевантных фрагментов → ответ в чате.

### 7.2. Что не входит в MVP

В первую версию не входят:

- OCR;
- обработка сложных PDF;
- большие наборы документов;
- многопользовательский режим;
- аутентификация и разграничение доступа;
- production-ready безопасность;
- мониторинг и централизованное логирование;
- отдельный backend;
- отдельный frontend;
- Kubernetes;
- сложные ingestion-пайплайны.

## 8. Ограничения и риски

### 8.1. Ограничения

- запуск только на обычных ноутбуках без GPU;
- ограниченные CPU и RAM;
- расчет на одного пользователя;
- только небольшие документы;
- демонстрационный, а не production-сценарий.

### 8.2. Основные риски

- **Медленный inference.** На слабых ноутбуках ответы могут генерироваться заметно медленно.
- **Ограничение по памяти.** При слишком большом контексте или неудачной конфигурации система может работать нестабильно.
- **Слабая работа со сложными PDF.** Документы с таблицами, картинками и плохим текстовым слоем могут разбираться плохо.
- **Ограничения маленькой модели.** Качество ответов может быть ограничено по глубине и точности.

## 9. Критерии готовности

MVP считается готовым, если выполнены все условия ниже:

1. Модели скачиваются в локальную папку.
2. `docker compose up -d` поднимает все нужные сервисы.
3. Open WebUI открывается локально в браузере.
4. Chat model отвечает на обычные вопросы.
5. Embedding model используется для индексации документов.
6. Пользователь может загрузить небольшой документ.
7. Пользователь может задать вопрос по документу и получить осмысленный ответ.

## 10. Этапы реализации

### Этап 1. Подготовка каркаса проекта
Нужно подготовить минимальную структуру репозитория, `docker-compose.yml`, `.env.example` и базовые каталоги.

### Этап 2. Подготовка загрузки моделей
Нужно добавить скрипт скачивания chat-модели и embedding-модели в локальную папку.

### Этап 3. Поднятие локального чата
Нужно запустить `llama-chat`, подключить его к Open WebUI и проверить обычный чат без RAG.

### Этап 4. Подключение embeddings и проверка RAG
Нужно запустить `llama-embed`, подключить его к Open WebUI и проверить вопрос-ответ по загруженному документу.

## 11. Итог

На текущем этапе фиксируем следующий технический замысел:

- локальный RAG-сервис для учебного хакатона;
- `Open WebUI` как пользовательский интерфейс;
- два инстанса `llama.cpp`: отдельно для chat и embeddings;
- `Qwen/Qwen3-1.7B-Instruct-GGUF` как основная LLM;
- `Qwen/Qwen3-Embedding-0.6B-GGUF` как embedding-модель;
- запуск через Docker Compose;
- ориентация на обычный ноутбук без GPU;
- минимальный и воспроизводимый MVP без лишней инфраструктуры.
